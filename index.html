Understanding the difference
Why the Flutter bar fluctuates
and iOS never does
1
The root cause
Flutter lives in one process. iOS rendering lives in another.

Every Flutter app runs inside a single process: your Dart business logic, your widget tree, and the Skia graphics engine all share one pool of CPU time. When any one of them gets busy, the others wait.

iOS Core Animation runs its render server — called backboardd — as a completely separate OS process at higher system priority than your app. It cannot be interrupted by anything your app does.

Flutter architecture
→Your app + render engine = same process
→CPU spike in your code → raster thread waits
→Raster thread waits → frame arrives late
→Frame arrives late → user sees jank
iOS Core Animation
→Your app + render server = different processes
→CPU spike in your code → render server unaffected
→Render server runs at higher OS priority
→FPS stays at 60 regardless of app load
2
The pipeline difference
Flutter fires 8 steps every frame. iOS fires 5 — and the last 3 happen outside your app.

Every time anything changes on screen, Flutter runs this sequence entirely inside your app's process. Every step costs your CPU budget. Every step that runs late pushes the frame past the 16.67ms deadline.

Flutter — 8 steps, all in your process
1. setState() triggered Dart
2. Widget tree rebuilt Dart
3. Element reconcile Dart
4. RenderObject layout Dart
5. Skia paint() called Skia
6. drawRRect / drawCircle etc Skia GPU
7. Composite layers GPU
8. GPU flush → display GPU
iOS Core Animation — 5 steps, last 3 leave your app
1. @State change triggers SwiftUI diff Swift
2. CALayer transaction committed CA
3. Render server receives transaction ← leaves your app backboardd
4. GPU composites layers — off-process OS GPU
5. Display — your app thread already free Display
Steps 3–5 in iOS happen in a separate OS process. Your app thread is already free to do other work while the frame is being composited. In Flutter, steps 5–8 still use your process. That is the entire explanation for why the bar fluctuates.
3
What a CPU spike actually costs
Any spike in your app — network, database, image decode — steals from Flutter's frame budget.

The frame budget is 16.67ms (1 second ÷ 60 frames). Flutter has to fit all 8 pipeline steps inside that window. When your app does something expensive — parsing a JSON response, loading from SQLite, decoding a large image — it eats into that budget before Skia even starts drawing.

In iOS, the same work happens entirely in your app thread. The render server does not share that thread. The network call can block for 50ms, and the render server will still hit every 16.67ms deadline without missing a single frame.

16.67
ms — total frame budget at 60fps. Flutter must fit 8 steps here.
~8
ms — average Flutter build + raster on a modern iPhone. Leaves 8ms buffer.
1
ms spike is enough to push a tight frame over budget and drop it.
0
ms — CPU spike cost to iOS Core Animation. Render server is immune.
Real-world culprits that spike Flutter CPU: JSON.decode() on large payloads, SQLite queries on main thread, image.network() decode, setState() in a high-frequency callback (scroll listener), platform channel round-trips.
4
What this costs your product
Users don't count frames. But they feel them — and they leave.

The human eye cannot consciously detect 60fps vs 58fps. But it can detect accumulated jank — the sticky feeling when a list scroll hesitates for 80ms, or an animation stutters during a network call. Below are documented industry patterns that trace back to frame drops.

Flutter — real risks at scale
→Network response on main thread drops frames during parse
→Dart GC pauses can stall the raster thread
→setState() on a high-freq timer triggers full rebuilds
→Image.network() decodes on UI thread by default
→Mitigations exist — isolates, compute(), RepaintBoundary — but require deliberate engineering
iOS — what you get for free
→URLSession network calls never touch the render thread
→No GC — ARC is deterministic, no stop-the-world pauses
→Core Animation runs at higher priority than your app
→AsyncImage decodes off-thread automatically
→Smooth scrolling, haptics, Face ID — OS handles everything
5
The trade-off — honest answer
Flutter made this trade deliberately. It is the right choice for some products and the wrong choice for others.

Flutter draws everything itself so that one codebase produces identical pixels on iOS and Android. One team, one app, one release cycle. For products targeting both platforms, that is a genuine and significant cost saving. The jank risk is manageable with correct engineering patterns.

SwiftUI wins on iOS native quality: system components, haptics, iCloud Keychain autofill, MetricKit, Face ID — all free, all perfect. But it is iOS-only. A separate Android team is required.

The one-sentence version for management
Flutter draws — it owns a canvas and paints every pixel itself, which makes the FPS bar sensitive to anything your app does. SwiftUI delegates — it tells the OS what changed and hands rendering to a separate system process, which is why the bar never moves. The fluctuating bar is not a bug. It is the price Flutter pays for being everywhere.
